**The Triton 3.6 DSL is at its structural ceiling for this operator.** Remaining unexplored direction: **cluster (num_ctas) cooperative reduce via DSMEM** — the only path that genuinely captures the "one-launch" win the persistent kernel and atomic-last-writer attempts failed to deliver. Triton 3.6 cluster on Blackwell is reportedly immature; this lever appropriately belongs in a separate-campaign DSL switch to CuTe-DSL or raw CUDA where cluster + DSMEM are first-class.

## Open levers for next round

Round-2 sub identified (per `variants/triton_pdl_grid_dispatch/kernel.py` header, section 5):

1. **Persistent kernel** — each CTA owns a full (b, h_kv) and streams all KV, no inter-CTA reduce. Loses on B=1 (only 8 CTAs natural parallelism) unless paired with intra-CTA work-stealing across (b, h_kv) tuples. Substantial refactor; modest expected gain over 1.44x.
2. **GQA packing (4 head groups per CTA)** — flagged AGAIN as lower leverage (NCU SM Compute 33% vs DRAM 57% — memory is still the binding constraint).
3. **Reduce reg pressure** (NCU 92 reg/thread, achieved occupancy 29.5% vs theoretical 31.25%) — "Est. Local Speedup 68.75% if occupancy maxed." Would need a structural rewrite (smaller acc tile, f16 acc, or pack-4-GQA). Hard to action standalone.
4. **Cluster (num_ctas) cooperative reduce via DSMEM** — Triton 3.6 cluster on Blackwell reportedly immature; needs smoke probe first. Alternatively switch DSL to CuTe-DSL or raw CUDA where cluster + DSMEM are first-class.

Round-2 also CLOSED the original "kill the second launch" lever via the atomic-last-writer path (regressed -10-15%). PDL achieves similar latency-hiding goal without restructuring. The DSMEM cluster path remains open as the genuine "one-launch" alternative.

Round-3 closed two B=16-routing dead-ends (fused-for-B16; BLOCK_N=32-for-B16-split). Sub flagged a fresh untried lever that resolves the B=16 bimodal-distribution trap: **dynamic SPLIT inside the kernel**.

Round-4 attempted that lever and partially landed it as `triton_per_workload_split_max` (forensic sub-anchor; real +27%/+20% on the 2 long-KV B=16 workloads via per-workload SPLIT_MAX routing on `kv_indices.shape[0] // batch_size` host-shape — distinct from round-1's closed `.item()` peek). Headline ceiling only ~1% because long-KV is 2/48 workloads. Round-4 also closed: **SPLIT_MAX=8 globally for B=16** (-3.5%; reduce partial-O load doubles, inflation > gain — fixed by per-workload routing in iter-3); **short-KV-only fused-B16** (-24% B=16; per-batch kv_len variance breaks load balance within short-KV workloads — fused-B=16 dead at every routing granularity short of per-batch dispatch); **reduce num_warps 2→4** (-3.22%).

Remaining unexplored levers after round 5:
1. **Packed reduce kernel** — pack GQA=4 q-heads per reduce CTA. NCU "Est. Local Speedup 97.24%" but round-4 wave-time analysis suggests gain may be marginal on B=16 short-KV (already 0.86 waves) and only modest on B=64. Triton-internal, still untried.
2. **Cluster (num_ctas) cooperative reduce via DSMEM** — would eliminate PARTIAL_O HBM round-trip entirely. **This is now the only path that genuinely captures the "one-launch" win** the persistent kernel was supposed to deliver. Triton 3.6 cluster on Blackwell reportedly immature; either needs a Triton-3.6 cluster smoke probe first OR a DSL switch to CuTe-DSL / raw CUDA where cluster + DSMEM are first-class.
3. **f16/bf16 accumulator on partials** — halves HBM traffic on partial reads/writes. Tolerance-risky; bf16 outputs already at 1e-2 to 1e-3 abs_err.

**Closed by round 5**:
- **Persistent kernel** in Triton — three structural reasons (compiler limit on `return` in `while`; pipeliner doesn't carry across outer-while; spin-wait at grid=num_SMs stalls SM). The lever is functionally closed for Triton 3.6 (would need a fundamental compiler-level pipeliner extension to revisit). Wall-time math also showed the only available win was the reduce-launch tax (~1-2us, max ~10% headline on B=16), and capturing it needs in-kernel sync which has failed twice (r2 atomic / r5 spin-wait).

**Closed by round 6**:
- **Packed reduce kernel** in Triton — CTA-count → wave-count → latency-hiding mechanism. Packing reduces CTA count below 1 wave on B=16 small reduce kernel; warp-scheduler can no longer hide per-CTA latency across waves. NCU "Est. Local Speedup" pointed in the wrong direction (it assumes occupancy increase, not CTA-count decrease). Plateau-at-parent across num_warps ∈ {2,4,8}. (NCU pitfall now documented in `templates/skills/profiler-ncu/ncu.md`.)
