# Harness Ledger

Append-only timeline of master-evaluated harness modification proposals. One line per decision.

## Format

Each entry is a markdown list item — a leading `- `, added automatically by
`append_ledger` — so the file renders as a list instead of collapsing into one
run-on paragraph in a markdown preview. The logical content of one entry:

```
- YYYY-MM-DD round-N edit-id scope: short rationale (accepted|rejected: reason)
```

- **edit-id**: short stable handle, e.g. `bench-modes-add-warmup` or `task-md-clarify-prior-block`.
- **scope**: file path that the edit touches (relative to AKO4X root). E.g. `templates/skills/bench/SKILL.md` or `scripts/run_local.py`.
- **rationale**: 1-2 sentences. For accepted: why it's a real harness gap, not generic knowledge. For rejected: why it failed the gate.

### Mode-2 round-summary line (no proposals solicited)

In a Mode-2 campaign phase-2 never runs, so a round produces no per-proposal entries — instead exactly one round-summary line. It intentionally omits the `edit-id scope:` and `(accepted|rejected: reason)` fields above (there is no proposal to accept or reject). Forms:

```
- YYYY-MM-DD round-N <variant-name-or-no-change> (mode-2; no-proposals-solicited; kernel_changed=<T|F>; archived=<T|F>)
- YYYY-MM-DD round-N - (mode-2; phase1-<crash|timeout>: <reason>)
```

When `archived=F` is a pre-archive gate rejection (silent-skip or library-delegation), append the reason inside the parenthetical — e.g. `… kernel_changed=T; archived=F; library-call-suspected: flashinfer` — mirroring how the failure form carries `phase1-<crash|timeout>: <reason>`.

A grep for `scope:` or for the `accepted`/`rejected:` token will therefore NOT match Mode-2 lines — filter Mode-2 rounds by the `mode-2` tag instead.

## Reject reasons (typical taxonomy)

- `rejected: generic-knowledge` — even granting the evidence, the lesson is general practice any capable agent already brings; not something this benchmark/harness taught this session, so recording it wouldn't change future behavior.
- `rejected: out-of-scope: task-identity` — touches a FROZEN task-identity item; canonical list in `templates/closed-loop-scope.md`.
- `rejected: out-of-scope: bench-frozen-behavior` — touches a frozen-for-comparability item; canonical list in `templates/closed-loop-scope.md` (benchmark-bound details in the active benchmark SKILL's `## Frozen for bench comparability`).
- `rejected: out-of-scope: master-infra` — touches anything under `master/` (master.py / MASTER.md / harness-ledger.md); privilege boundary, not a per-campaign rule.
- `rejected: silent-skip-suspected: <pattern>` — phase-1 transcript matched a known TRAPS.md pattern or general silent-bug heuristic; variant not archived AND proposal rejected to avoid normalizing the cheat.
- `rejected: library-call-suspected: <lib>` — the resolved archive artifact delegates the operator's computation to a precompiled vendor kernel (FlashInfer / DeepGEMM / cuBLAS / cuDNN) instead of implementing it; variant not archived so it can't become the family anchor. Canonical rule: the active benchmark SKILL's `## Valid solution: write your own kernel` section. Orthogonal to the cheat-check (a library call recomputes honestly, so it passes that). In Mode 2 the reason rides inside the `archived=F` round-summary parenthetical, not a per-proposal entry.
- `rejected: insufficient-evidence` — proposal cites no concrete trajectory artifact (ITERATIONS.md line / commit / bench output).
